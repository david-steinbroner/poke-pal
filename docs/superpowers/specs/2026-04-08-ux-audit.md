# Poke Pal v0.5.2 UX Audit

**Date**: 2026-04-08
**Auditor**: UX Specialist
**Target device**: iPhone 375px (primary), any mobile browser
**Live URL**: https://poke-pal.pages.dev

---

## Executive Summary

Poke Pal has solid bones: fast static pages, a clear 3-tab nav, and the core copy-paste loop works. But the app currently treats every element with equal visual weight, buries the primary action on most pages, and wastes precious above-the-fold real estate on explanatory text that active Pokemon GO players do not need. The result is an app that looks like a reference site instead of a tool you grab mid-battle.

The biggest wins are: (1) making the search input the hero on the home page, (2) collapsing header chrome on every page, (3) making CopyBar feel like THE thing you came for, and (4) removing instructional copy that patronizes the target audience.

## 1. Page-by-Page Teardown

### 1A. Home Page (`src/app/page.tsx`)

**User goal**: "I'm about to battle. I need counters for a specific Pokemon, or I want to see what's meta in the current cup."

**Problems**:

1. **The h1 is wrong and too loud** (line 17). "Pokemon GO PvP Search Strings" describes the product category, not what the user needs right now. It takes up prime vertical space and tells the user nothing actionable.
   - **Fix**: Remove the h1 entirely, or reduce it to `text-xs text-muted-foreground` and change to "Poke Pal". The app name is already understood from the tab title and OG tags. The search input IS the headline.

2. **The subtitle is filler** (line 19). "Find counters, build teams, copy strings. Paste in GO." is a marketing tagline, not in-app guidance. Users who are on this page already know what the app does.
   - **Fix**: Delete lines 18-20 entirely. Replace with nothing. The search input placeholder does this job.

3. **Search input is visually equal to everything else** (line 23, search-input.tsx line 69). At `min-h-11` with default Input styling, it blends into the page. On a 375px screen, the user sees: version tag, h1, subtitle, search input, "Quick Picks" label, chips -- all before scrolling. The search input should be the dominant element.
   - **Fix**: Increase search input to `min-h-[52px] text-lg` with a more specific placeholder: "Who are you fighting?" (action-oriented, specific to the use case). Add `autoFocus` on mount so the keyboard opens immediately on mobile -- this signals "type here first".

4. **Quick Picks section label is unnecessary** (line 27). "Quick Picks" is obvious from context. The label adds visual noise.
   - **Fix**: Remove the h2 "Quick Picks" label. The chips speak for themselves. If you keep a label, use a single-word label like "Popular" at `text-[11px]`.

5. **"Build a Team" section is a single orphan link** (lines 69-81). A full section header + one pill link is an odd pattern. It makes the page feel incomplete.
   - **Fix**: Move the Team Builder link into the "What's Live" row as another pill. Or remove it entirely -- the bottom nav already has a Teams tab. Duplicating nav destinations in-page wastes space.

6. **Three section headers competing at the same weight** (lines 26, 47, 70). All three are `text-sm font-medium text-muted-foreground`. When everything is the same weight, nothing stands out.
   - **Fix**: Remove section headers entirely. Use visual grouping (spacing) instead. The content types (search input, chips, chips) are already visually distinct.

**Ideal home page flow**: User lands -> sees search input with "Who are you fighting?" -> types "Giratina" -> taps result -> lands on counter page. Total time: 3 seconds. Secondary path: scrolls past search -> taps a Quick Picks chip or a live league pill.

**Recommended home page structure** (top to bottom, above fold):
- Search input (52px tall, autofocused, dominant)
- Quick pick chips (no label)
- Live league chips (no label, or a subtle "Live Now" dot indicator on each chip)

### 1B. Leagues Landing (`src/app/leagues/page.tsx`)

**User goal**: "Which cups are live right now? Let me pick one to see the meta."

**Problems**:

1. **Back button + h1 row is redundant** (lines 29-37). The bottom nav already shows Leagues as active. A back arrow to "/" is unnecessary when "/" is one tap away in the bottom nav. This wastes 48px of vertical space.
   - **Fix**: Remove the back button. Keep just the h1 "Leagues" but reduce it to `text-lg font-semibold` to save vertical space.

2. **LeagueCard is too information-dense** (league-card.tsx lines 31-57). Each card shows: name, Live badge, CP cap, season label, meta count. On a 375px screen, the secondary info line wraps. "Season 27 GBL" means nothing to most players -- they just want to know which cup is live.
   - **Fix**: Simplify to: league name (left), "Live" or start date (right), CP cap on second line only. Drop the season label and meta count from the card -- they're visible on the league detail page.

3. **"Coming Up" section with 50% opacity** (league-card.tsx line 35, `opacity-50`). Dimming the entire card is too aggressive. Users scanning the page might think these are disabled or broken.
   - **Fix**: Use `opacity-70` or better yet, skip opacity entirely and just use a subtle visual distinction: no green "Live" badge, a `text-muted-foreground` start date on the right side. The absence of "Live" already communicates "not live".

4. **No empty state is engaging** (lines 60-65). "No active cups right now. Check back when the next GBL rotation starts." is passive. It does not tell the user what to do instead.
   - **Fix**: "No active cups right now. Try building a team for Great League." with a link to `/teams?l=great-league`.

### 1C. League Detail (`src/components/league/league-page-client.tsx`)

**User goal**: "Show me the meta for this cup so I can pick my team and copy the search string."

**Problems**:

1. **Back button is redundant** (line 71). Same issue as leagues page -- bottom nav handles this.
   - **Fix**: Remove the back link. Saves ~40px vertical space.

2. **Header metadata is too verbose** (lines 77-87). The page shows: league name, "Season 27 GBL . CP 1,500 . Updated Apr 7", then potentially type restrictions. Three lines of metadata before any actionable content.
   - **Fix**: Collapse to one line: `h1` league name + inline CP cap badge. Move "Updated Apr 7" to the page footer or remove it entirely -- users don't care when you updated the meta, they care that it IS current. The season label adds no value.

3. **"Find meta Pokemon you own" label above CopyBar** (lines 90-93). This label is instructional and unnecessary. The CopyBar's "Copy" button already communicates its purpose.
   - **Fix**: Remove the label text. Let the CopyBar stand alone. If anything, change the truncated search string preview to show a summary like "All 14 meta picks + CP 1500" instead of the raw string.

4. **The CopyBar appears before the tier list** (line 93). Users who land here want to browse the meta FIRST, then copy. The copy bar for "all meta" is a secondary action -- the primary action is browsing S/A/B/C tiers and adding to your team.
   - **Fix**: Move the full-meta CopyBar below the TierAccordion, or collapse it into a single "Copy all meta" text button at the top. The tier accordion should be the first interactive content below the header.

5. **Team section only appears after adding a Pokemon** (lines 96-105). The user sees zero indication that they can build a team until they tap a + button on a meta card. The + button on MetaPokemonCard is 44x44 but has no label.
   - **Fix**: Show a lightweight empty team state above the tier accordion at all times: three dashed circles with "Tap + to build your team" in muted text. This teaches the affordance without tooltips.

6. **TierAccordion labels are long** (tier-accordion.tsx line 13-17). "S Tier -- Meta Defining" takes a lot of horizontal space. On 375px, these labels wrap.
   - **Fix**: Shorten to just "S Tier (4)" or even "S (4)". The descriptors ("Meta Defining", "Strong Picks") are editorial opinion and add clutter.

### 1D. Counter Page (`src/app/counter/[pokemon]/page.tsx`)

**User goal**: "I'm fighting [Pokemon]. What beats it? Let me copy the search string."

This is the strongest page in the app. The user goal and page structure align well. Issues are minor.

**Problems**:

1. **Type badges and effectiveness section take a lot of vertical space** (lines 55-113). The "Weak to" / "Resists" section with colored type badges is useful reference but pushes the CopyBar (the primary action) below the fold on small screens.
   - **Fix**: Move the CopyBar directly after the h1 + type badges row (after line 64), before the effectiveness section. The user's first instinct is "copy the counter string." Effectiveness details are reference material they'll read after copying.

2. **CopyBar is not visually distinct enough** (copy-bar.tsx). The `bg-muted/50` background and `rounded-lg border` make it look like every other card on the page. The primary CTA ("Copy") is the right element, but the bar itself doesn't scream "this is why you're here."
   - **Fix**: Make the CopyBar more prominent app-wide: use `bg-primary/5 border-primary/20` (a subtle brand-tinted background). Increase the Copy button to `min-h-[44px] px-5 text-base font-semibold`. This makes the CopyBar the visual anchor of every page it appears on.

3. **"Build a team around X (Great League)" link** (lines 175-180). Hardcoded to Great League. If the user came from Ultra League, this link is wrong.
   - **Fix**: Pass league context via a query param or referrer. If no league context, show multiple options: "Build a team in Great League | Ultra League" as two compact links.

4. **Shadow Variant section** (lines 166-173). This appears at the very bottom. Many users won't scroll this far, and shadows are important in PvP.
   - **Fix**: Integrate shadow info into the main counter cards. Add a small "Shadow" badge to counters that have shadow variants, rather than a separate section.

5. **PokemonCard has no tap target** (pokemon-card.tsx). Counter cards are display-only -- you can't tap them to see that Pokemon's counter page. This is a dead end.
   - **Fix**: Wrap the entire PokemonCard in a `Link` to `/counter/{pokemonId}`. Add a subtle chevron-right icon. This creates the exploration loop: I'm fighting Giratina -> Giratina is countered by Togekiss -> what counters Togekiss?

### 1E. Teams Page (`src/app/teams/page.tsx`)

**User goal**: "I want to build a 3-Pokemon team for a specific league and get the search string."

**Problems**:

1. **Same redundant header pattern** (lines 204-215). Back link + h1 + subtitle = 3 lines of chrome before any interactive content.
   - **Fix**: Remove back link (bottom nav), remove subtitle "Build a team, check coverage, copy the search string" (obvious from context). Just: "Team Builder" as h1.

2. **LeaguePicker position is good** (line 217). This is correctly placed as the first interactive element. No change needed.

3. **Empty slot cards are passive** (lines 219-241). Three empty slots with "Lead", "Mid", "Closer" labels. When empty, they show "Add" buttons but don't suggest what to add.
   - **Fix**: When all three slots are empty, show the meta suggestions inline: "Popular in Great League:" followed by the top 3 S-tier picks as quick-add chips. Reduce the empty state friction.

4. **"Share team link" is too hidden** (lines 263-271). It's a tiny `text-xs` link with a 14px icon. Sharing is a growth feature -- it should be more prominent.
   - **Fix**: Make it a full-width bordered button at `min-h-11`, placed right below the CopyBar. "Share this team" with the share icon.

5. **"Counter Details" section at the bottom** (lines 299-317). Linking to individual counter pages from the team builder is useful but the section header "Counter Details" is generic.
   - **Fix**: Rename to "See counters for" and keep the current chip links. Or better: make each TeamSlotCard link directly to the counter page when tapped, and remove this redundant section.

6. **The coverage chart, threat list, and suggestions compete** (lines 273-297). Three analysis sections stacked vertically. On mobile, this is a lot of scrolling past analytical data. Most users want: team -> search string -> done.
   - **Fix**: Collapse coverage chart and threat list into an expandable "Analysis" accordion, default collapsed. Keep suggestions visible since they drive action.

### 1F. Bottom Nav (`src/components/bottom-nav.tsx`)

**Good**: 3 tabs is the right number. Icons are appropriate. Active state is clear. Touch targets are adequate (`pt-2.5 pb-1.5` with full flex-1 width). Safe area inset is handled.

**Problems**:

1. **"Search" label for the home tab** (line 8). The home page is more than search -- it's the landing page with quick picks and live leagues. But functionally, search IS the primary action, so this label works. No change needed.

2. **No visual indicator of activity** (e.g., team count badge on Teams tab). If the user has a team in progress, there's no reminder.
   - **Fix**: Add a small dot indicator on the Teams icon when `localStorage` has a saved team. This is a low-priority enhancement.

## 2. Information Hierarchy Problems

### Global issues across all pages:

1. **Every h2 is `text-sm font-medium text-muted-foreground`**. Section headers on home, leagues, counter, and teams pages all use the exact same styling. When section labels look identical to body text metadata, the page has no visual rhythm.
   - **Fix**: Either remove section labels entirely (let spacing do the work) or differentiate them: `text-xs font-semibold uppercase tracking-wider text-muted-foreground` for section labels, keeping body meta as `text-sm text-muted-foreground`.

2. **All pill-style links have the same styling** (home page quick picks, league pills, team builder link). They're all `rounded-full border px-3 py-1.5 text-sm`. There's no visual distinction between "navigate to a Pokemon" and "navigate to a league."
   - **Fix**: Use different chip styles for different content types. League chips could have a subtle colored left border matching the league. Pokemon chips stay neutral. Team builder could be a filled button, not a chip.

3. **CopyBar competes with page content**. On league detail and counter pages, the CopyBar is visually identical to other bordered cards. It should be the loudest element on the page because copying IS the product.
   - **Fix**: As described in 1D, give CopyBar a distinct background tint and a larger copy button. It should be the only element on each page with a filled primary-color button.

4. **Version banner steals the first pixels** (layout.tsx line 54-56). `v0.5.2` in the top-right corner is the first thing rendered in the layout, before any page content. It takes `pt-1` + the text height (~20px total).
   - **Fix**: Move the version to the bottom of the page (in a footer area), or remove it from the layout entirely and only show it in a settings/about context. No user needs to see "v0.5.2" on every page load.

## 3. Mobile-First Issues (375px iPhone)

1. **CopyBar truncation is correct** but the `code` element uses `truncate` (copy-bar.tsx line 26-27). On 375px, after accounting for padding (16px x 2) and the Copy button (~68px), the code element gets about 220px. Long search strings show as `giratina-altered,medicham,...`. The user has no idea what the full string contains.
   - **Fix**: Below the CopyBar, add a `text-[11px] text-muted-foreground` summary: "5 counters for Giratina" or "14 meta picks + CP 1500". This gives meaning to the opaque search string.

2. **Empty team slot circles are 28x28px** (inline-team-section.tsx line 166, `h-7 w-7`). These are too small to be meaningful and too small to tap.
   - **Fix**: Increase to `h-9 w-9` (36px) or replace with a dashed-border pill that says "Add" matching the filled team chip height.

3. **Accordion triggers on league pages** (tier-accordion.tsx). The default AccordionTrigger from shadcn uses chevrons. On 375px, a long label like "S Tier -- Meta Defining (4)" plus a chevron icon can wrap to two lines.
   - **Fix**: Shorten labels as mentioned in 1C. Ensure `AccordionTrigger` has `min-h-[44px]` for tap target compliance.

4. **Suggestion chips in inline team section** (inline-team-section.tsx lines 174-188). "Try:" label + 3 chips with `Plus` icons and Pokemon names. On 375px, these can wrap to 2-3 lines, pushing the CopyBar further down.
   - **Fix**: Limit to 2 suggestions instead of 3 on mobile. Or use a horizontal scroll container (`flex-nowrap overflow-x-auto`) instead of `flex-wrap`.

5. **Bottom nav safe area** (bottom-nav.tsx line 18). The `pb-[env(safe-area-inset-bottom,0px)]` is correct, but `pb-20` on the main content (layout.tsx line 57) may not be enough on all devices. iPhone 14 Pro has a 34px home indicator.
   - **Fix**: Increase to `pb-24` to ensure content is never hidden behind the nav + home indicator on any iPhone model.

## 4. Missing "Ah Ha" Moments

### When does the user currently feel "this is useful"?

The magic moment is: copy a search string -> open Pokemon GO -> paste it -> see exactly the Pokemon they need. This happens AFTER they leave the app. Inside the app, the moment is pressing "Copy" and seeing the "Copied! Paste in Pokemon GO" toast.

### How to get there faster:

1. **Home page should auto-focus the search input** (search-input.tsx). Zero taps to start typing. The keyboard should open on load. This cuts one interaction from the critical path.
   - Add `autoFocus` to the Input in search-input.tsx line 62.

2. **After copying, show a visual confirmation that persists longer**. The current toast disappears quickly. The CopyBar button says "Copied!" for 2 seconds (copy-bar.tsx line 16). Consider keeping the "Copied!" state for 4 seconds and adding a checkmark icon.

3. **The team builder's "ah ha" is the coverage score** (inline-team-section.tsx line 122, `{analysis.coverageScore}/18`). But it's a tiny muted number in the header row. Most users won't notice it.
   - **Fix**: Make the coverage score a prominent visual element: a small circular progress indicator or a bold number that animates when it changes. "14/18 types covered" as `text-base font-bold` with a color gradient (red at 6, yellow at 12, green at 16+).

4. **Missing: team role assignment callout**. When the user adds a second Pokemon, roles auto-assign (lead/swap/closer). This is genuinely novel -- no other tool does this. But the role labels are `text-[10px]` on team chips (inline-team-section.tsx line 138). They're nearly invisible.
   - **Fix**: When roles first appear (team goes from 1 to 2 Pokemon), show a brief toast: "Roles assigned: Bastiodon leads, Medicham swaps". Make the role labels `text-xs` and give them more visual weight -- colored pill backgrounds instead of just colored text.

5. **Missing: what to do after copying**. The toast says "Copied! Paste in Pokemon GO" but there's no deep link or visual guide. Consider adding a one-time helper below the CopyBar: "Open GO -> Search bar -> Paste" with simple icons. Show this only on first copy, then never again (localStorage flag).

## Design Philosophy Note

This app is a tool, not a content site. Every pixel should serve the copy-paste loop:

1. User has a question ("what beats Giratina?" or "what's meta in Fantasy Cup?")
2. User gets the answer (counter list or meta tier list)
3. User copies the search string
4. User pastes in Pokemon GO

Anything that does not directly support steps 1-4 is visual debt. The current version explains too much and shows too little. Strip the chrome, enlarge the actions, trust the user to understand a search bar and a copy button.
